# Deployment Guide — agent-suite

This guide walks a new operator through deploying the agent suite from
scratch on a Linux machine. It covers every step needed to go from a bare
box to a running, verified suite.

**Audience:** an operator or collaborator onboarding into the suite. No prior
knowledge of the internals is assumed — if a step requires a decision, the
guide explains the trade-off.

**Time estimate:** 30–60 minutes (mostly Postgres + package installation).

---

## 1. Architecture in one paragraph

The suite is six components that share one Postgres store:

- **regista** (Tier 0) — the append-only, hash-chained, signed event store.
  Every other component is a client of regista.
- **agent-notes** (Tier 1) — the agent face: CLI + skills for breadcrumbs,
  memories, reflections, work-item lifecycle. Has its own database for
  projections (pgvector search, op_log) but writes events through regista.
- **agent-provenance / cairn** (Tier 1) — the compliance keystone: intercepts
  harness tool calls and attests them into regista as signed events.
- **dossier** (Tier 1, optional) — the human face: web UI.
- **acb** (Tier 2, optional) — capability/credential parity across harnesses.
- **agent-wake** (Tier 2, optional) — external→session signaling.

**Tier 0–1 is the irreducible core.** Tier 2 is optional for a first
deployment.

## 2. Prerequisites

### 2.1 External dependencies (you provide)

| Dependency | What it is | Why it's needed |
|-----------|------------|-----------------|
| **Postgres 15+** | A reachable Postgres instance | The store. Can be on the same machine or a remote host. |
| **Python 3.12+** | On the deployment box | All components are Python. 3.14 is fine. |
| **uv** | Python package manager (`pipx install uv`) | Installs the components as CLI tools. |
| **A signing key** | Ed25519 keypair per principal | Non-repudiation. Generated by `regista provision-principal`. |
| **A harness** | Claude Code, opencode, or Hermes | The agent harness whose tool calls cairn attests. |

### 2.2 Postgres setup

The suite needs **two databases** on the Postgres instance:

1. **`regista`** — the event store. Owned by a service role (e.g.
   `regista_app`). Contains per-project schemas (one per project slug).
2. **`agent_notes`** — agent-notes' projection database. Owned by a service
   role (e.g. `agent_notes_app`). Contains memories, links, op_log, etc.

If you're sharing an existing regista store (e.g. a team deployment), you
only need to create the `agent_notes` database — the regista database is
already there.

To create both from scratch:

```sql
-- On the Postgres host (as superuser)
CREATE ROLE regista_app WITH LOGIN PASSWORD '<choose-a-password>';
CREATE DATABASE regista OWNER regista_app;
GRANT ALL PRIVILEGES ON DATABASE regista TO regista_app;

CREATE ROLE agent_notes_app WITH LOGIN PASSWORD '<choose-a-password>';
CREATE DATABASE agent_notes OWNER agent_notes_app;
GRANT ALL PRIVILEGES ON DATABASE agent_notes TO agent_notes_app;
```

### 2.3 Install uv

```bash
# On the deployment box
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc  # or restart shell — uv is at ~/.local/bin/uv
```

## 3. Install the components

Each component is installed as an isolated CLI tool via `uv tool install`.
This keeps their dependencies separate — no venv conflicts.

### 3.1 From local repos (if you have the source)

If you have the repos checked out locally (e.g. under `/projects/`):

```bash
# regista — the store (needs PyNaCl for Ed25519 signing, which is now a
# core dependency, but if installing from an older revision add --with pynacl)
uv tool install --python python3 /path/to/regista

# agent-notes — the agent face
uv tool install --python python3 /path/to/agent-notes

# cairn — provenance attestation
uv tool install --python python3 /path/to/agent-provenance

# agent-suite — the orchestration layer
uv tool install --python python3 /path/to/agent-suite
```

### 3.2 From public git (once repos are public)

```bash
uv tool install --python python3 "git+https://github.com/hraedon/regista.git@main"
uv tool install --python python3 "git+https://github.com/hraedon/agent-notes.git@main"
uv tool install --python python3 "git+https://github.com/hraedon/agent-provenance.git@main"
uv tool install --python python3 "git+https://github.com/hraedon/agent-suite.git@main"
```

### 3.3 Verify the CLIs are on PATH

```bash
export PATH="$HOME/.local/bin:$PATH"
# Add to ~/.bashrc for persistence:
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc

regista --version
agent-notes --version
cairn --version
agent-suite --help
```

## 4. Configure suite.env

The suite reads all shared config from one file: `~/.config/agent-suite/suite.env`.
The `agent-suite` CLI automatically loads this file before every command —
you don't need to `source` it manually.

> **System vs per-user:** The system file (`/etc/agent-suite/suite.env`)
> holds shared facts (DSN host, secret-backend pointers). The per-user file
> (`~/.config/agent-suite/suite.env`) holds each human's principal_id and
> default project. For a single-operator box, use the per-user file for
> everything.

### 4.1 Create the config

```bash
mkdir -p ~/.config/agent-suite
cat > ~/.config/agent-suite/suite.env << 'EOF'
# --- Spine (regista) ---
# The DSN for the regista event store. Use the service role, not a superuser.
REGISTA_DSN=postgresql://regista_app:PASSWORD@db-host:5432/regista

# Path to the store-level signing key (HMAC + principal index).
# Use a file: path for a single-operator box, or vault:/akv:/wincred: for team.
REGISTA_KEY_PATH=/home/USER/.config/regista/keys.json

# The project to provision during bootstrap.
REGISTA_PROJECT=agent_notes

# --- agent-notes native DB ---
# agent-notes uses its own database for projections. If this is the same
# Postgres host, point to the agent_notes database.
AGENT_NOTES_DSN=postgresql://agent_notes_app:PASSWORD@db-host:5432/agent_notes
AGENT_NOTES_REGISTA_WRITES=1
AGENT_NOTES_PROJECT=agent_notes

# --- cairn (provenance) ---
CAIRN_PROJECT=agent_provenance

# --- Principal identity ---
# Your principal ID — must match the one provisioned (step 5.2).
PRINCIPAL_ID=your-principal-id
EOF
chmod 600 ~/.config/agent-suite/suite.env
```

### 4.2 DNS note

If DNS is unreliable on your box, use the IP address in DSNs instead of
hostnames. The suite does not currently retry or cache DNS lookups.

## 5. Provision the store and principal

### 5.1 Copy or create the store-level signing key

The store-level signing key (`keys.json`) is shared by all clients of the
same regista store. It contains the HMAC key for chain verification and an
index of provisioned principals.

**If joining an existing store:** copy `keys.json` from another machine
that already has it:

```bash
mkdir -p ~/.config/regista
# From the existing machine:
scp operator@existing-host:~/.config/regista/keys.json ~/.config/regista/
chmod 600 ~/.config/regista/keys.json
```

**If creating a new store:** `regista provision` (run in step 6) will create
the initial schema. You may need to create the key file separately — check
`regista provision --help`.

### 5.2 Provision your principal

Each human and agent gets its own Ed25519 keypair. Provision yours for each
project you'll write to:

```bash
# For the agent_notes project
regista provision-principal \
  --principal your-principal-id \
  --project agent_notes \
  --secret-backend file \
  --private-key-dir ~/.config/regista

# For the agent_provenance project (if you'll attest tool calls)
regista provision-principal \
  --principal your-principal-id \
  --project agent_provenance \
  --secret-backend file \
  --private-key-dir ~/.config/regista
```

This creates:
- An Ed25519 private key at `~/.config/regista/your-principal-id_ed25519.key`
- A public key registered in the store
- An entry in `keys.json` linking your principal to its key

## 6. Run the bootstrap

The bootstrap runs the documented install order idempotently:

```
secret-backend probe → Postgres probe → provision → faces → provenance
```

### 6.1 Dry-run first

```bash
agent-suite bootstrap --dry-run --tier 0-1
```

Review the plan. It should show:
- `probe_secrets: pending` — would probe the secret backend
- `probe_db: pending` — would probe Postgres
- `provision: pending` — would provision the project schema + principal
- `faces: pending` — would run `agent-notes install-harness`
- `provenance: pending` — would run `cairn install-harness`

### 6.2 Run for real

```bash
agent-suite bootstrap --tier 0-1
```

Expected output:

```
  probe_secrets      done           secret backend reachable
  probe_db           done           Postgres reachable
  provision          done           project agent_notes provisioned
  faces              done           agent-notes harness installed
  provenance         done           cairn harness installed

bootstrap: OK
```

### 6.3 Verify idempotency

```bash
agent-suite bootstrap --tier 0-1
```

The second run should report `already_done` for provision and `done` for
faces/provenance (re-install is a no-op). `bootstrap: OK` both times.

## 7. Wire the harness

The bootstrap installs harness wiring for all harnesses by default. If you
need to install for a specific harness (e.g. Hermes):

```bash
# cairn — attests tool calls into regista
cairn install-harness hermes

# agent-notes — installs skills + env into the harness config
agent-notes install-harness hermes
```

### What this does

**cairn `install-harness hermes`:**
- Writes a managed env block to `~/.hermes/.env` with `REGISTA_DSN`,
  `REGISTA_KEY_PATH`, `CAIRN_PROJECT`, `PRINCIPAL_ID`,
  `CAIRN_HARNESS_NAME=hermes`
- Installs the cairn observability plugin to
  `~/.hermes/plugins/observability/cairn/`

**agent-notes `install-harness hermes`:**
- Writes `REGISTA_DSN`, `REGISTA_KEY_PATH`, `AGENT_NOTES_DSN` into
  `~/.hermes/.env`
- Installs skills (breadcrumb, memory, reflection, etc.) into
  `~/.hermes/skills/`

### Other harnesses

```bash
# Claude Code
cairn install-harness claude
agent-notes install-harness claude

# opencode
cairn install-harness opencode
agent-notes install-harness opencode

# All at once
cairn install-harness all
agent-notes install-harness all
```

## 8. Generate SUITE.lock

The lock pins the known-good component versions:

```bash
cd ~  # or wherever you want SUITE.lock
agent-suite lock
```

This writes `SUITE.lock` in the current directory. Verify it:

```bash
cat SUITE.lock
```

The lock should show each installed component's version. `agent-suite doctor`
checks installed versions against the lock and reports drift.

## 9. Verify with doctor

```bash
agent-suite doctor
```

Expected output for a healthy Tier 0–1 deployment:

```
suite: NOT OK   ← see note below

  regista                OK
  dossier                absent
  agent-notes           OK
  agent-provenance      OK
  agent-capability-broker  absent
  agent-wake            absent

lock: matches
```

`absent` for Tier 2 components is not a failure — it means they're not
installed, which is expected for a Tier 0–1 deployment.

If `agent-notes` shows a failure, check the detail — the most common issues
are:

| Failure | Cause | Fix |
|---------|-------|-----|
| `dsn_reachable: fail` | `AGENT_NOTES_DSN` not set or wrong | Check suite.env has the correct DSN for the `agent_notes` database |
| `schema_up_to_date: fail` | Missing tables | Run `agent-notes-migrate --all` |
| `skills_installed: fail` | Skills not installed | Re-run `agent-notes install-harness <harness>` |
| `links_audit: fail` | Dangling links in store | Pre-existing data issue — see troubleshooting |
| `harness_wired: fail` | No harness manifest | Re-run `agent-notes install-harness <harness>` |

### JSON output

```bash
agent-suite doctor --json | python3 -m json.tool
```

## 10. Onboarding a new collaborator

When a new person joins the team and needs access to the suite:

### 10.1 Provision their principal

On any machine with access to the store:

```bash
regista provision-principal \
  --principal their-name \
  --project agent_notes \
  --secret-backend file \
  --private-key-dir /tmp/their-keys

regista provision-principal \
  --principal their-name \
  --project agent_provenance \
  --secret-backend file \
  --private-key-dir /tmp/their-keys
```

### 10.2 Give them the key files

Securely transfer:
- `keys.json` (the store-level signing key — same for all clients)
- `their-name_ed25519.key` (their personal Ed25519 private key)

They should place these at `~/.config/regista/` on their machine.

### 10.3 They follow this guide

From step 3 onward — install components, create suite.env (with their
`PRINCIPAL_ID=their-name`), run bootstrap, wire their harness.

### 10.4 Per-user overlay

Each collaborator's `~/.config/agent-suite/suite.env` should have:

```bash
# System-wide facts (shared, from /etc/agent-suite/suite.env):
REGISTA_DSN=postgresql://regista_app:...@db-host:5432/regista
REGISTA_KEY_PATH=/home/USER/.config/regista/keys.json
AGENT_NOTES_DSN=postgresql://agent_notes_app:...@db-host:5432/agent_notes

# Per-user (this file):
PRINCIPAL_ID=their-name
AGENT_NOTES_PROJECT=their-default-project
```

The `agent-suite` CLI automatically merges system + per-user config.

## 11. Troubleshooting

### "no module named 'nacl'"

PyNaCl is required for Ed25519 signing. It's a core dependency of regista
but if installing from an older revision, install it explicitly:

```bash
uv tool install --force --with pynacl /path/to/regista
uv tool install --force --with pynacl /path/to/agent-notes
uv tool install --force --with pynacl /path/to/agent-provenance
```

### "No Postgres DSN found"

agent-notes reads `AGENT_NOTES_DSN` first, then falls back to `REGISTA_DSN`
from suite.env. If both are missing, it raises this error. Check that
`suite.env` exists at `~/.config/agent-suite/suite.env` and contains the
DSN.

### "regista: failed" in doctor but all checks are OK

This was a bug where regista's doctor didn't emit a top-level `ok` field.
Fixed — regista now emits `ok: true` when all checks pass. If you see this
on an older regista, upgrade.

### "cairn: plugin source not found"

The cairn `integrations/hermes/` directory wasn't included in the wheel.
Fixed in the pyproject.toml — if you see this, reinstall cairn from a
current source.

### "agent-notes: skills_installed: fail"

The `skills/` directory wasn't included in the wheel. Fixed — if you see
this, reinstall agent-notes from a current source.

### Dangling links in store

If `agent-notes doctor` reports `links_audit: fail` with "Dangling links",
this is a pre-existing data issue in the shared store (memory links
pointing to deleted entities). It does not affect writes — it's a data
quality check. To clean up:

```sql
-- On the agent_notes database, identify dangling links
SELECT * FROM links WHERE to_entity_id NOT IN (
    SELECT id FROM memories
);
```

Consult with the store administrator before deleting.

### DNS resolution failures

If hostnames don't resolve, use IP addresses in DSNs. The suite doesn't
retry DNS. For a permanent fix, add entries to `/etc/hosts` or fix DNS.

### Bootstrap fails at "provision" with "no project configured"

Set `REGISTA_PROJECT` in suite.env:

```bash
echo 'REGISTA_PROJECT=agent_notes' >> ~/.config/agent-suite/suite.env
```

## 12. Next steps after deployment

1. **Start your harness** — e.g. `systemctl start hermes` or launch Claude
   Code. The cairn plugin will begin attesting tool calls automatically.

2. **Verify attestation** — after a few tool calls, check the store:

   ```bash
   regista events --project agent_provenance --limit 5
   ```

   You should see `tool_call_begin` and `tool_call_end` events attributed
   to your principal.

3. **Write a breadcrumb** — test the agent face:

   ```bash
   agent-notes breadcrumb file --text "Suite deployed on $(hostname)"
   ```

4. **Run the interop proof** (optional, for a team pilot):

   ```bash
   agent-suite doctor --json  # should show suite_ok: true (or known fails)
   ```

5. **Set up scheduled backups** (Plan 005):

   ```bash
   agent-suite schedule --backup --verify-restore
   ```

6. **Set up alerting** (Plan 005, needs agent-wake):

   ```bash
   agent-suite schedule --alert-check
   ```

## Appendix A: Quick reference

| Command | What it does |
|---------|-------------|
| `agent-suite bootstrap --tier 0-1` | Install the core suite (idempotent) |
| `agent-suite doctor` | Check all components' health |
| `agent-suite lock` | Generate/update SUITE.lock |
| `agent-suite verify-restore` | Verify a restored store is intact |
| `agent-suite upgrade --check` | Check for available component upgrades |
| `regista provision --project <slug>` | Provision a project schema |
| `regista provision-principal --principal <id> --project <slug>` | Provision a principal keypair |
| `regista doctor --json` | Check regista health |
| `regista events --project <slug> --limit N` | Show recent events |
| `cairn install-harness <target>` | Wire cairn into a harness |
| `cairn doctor --json` | Check cairn health |
| `agent-notes install-harness <target>` | Install skills + env for a harness |
| `agent-notes doctor --json` | Check agent-notes health |
| `agent-notes breadcrumb file --text "..."` | Write a breadcrumb |

| File | Location | Purpose |
