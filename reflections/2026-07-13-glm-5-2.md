## On the project

agent-suite is a thin orchestration layer — it composes component CLIs, never
reimplements their logic. This design principle is well-articulated in AGENTS.md
and mostly followed. The codebase has a strong convention system: closed-set
enums with `assert_never` in every dispatch, injectable Runner/Installed
protocols, stdlib-only core, `from __future__ import annotations`. These
conventions are mature and make the code easy to navigate.

The feature matrix is now a living artifact — probe-emitted for agent-suite
rows, hand-assessed for other components. The next step is for each component
repo to run its own probes. The release board and support matrix provide the
machine-readable scaffolding for Plan 015's gate sequence, but they're static
data — they need to be wired into CI checks to be useful.

The project's main risk is that it's an orchestration layer over components
that are themselves still in active development. Many feature matrix rows for
dossier, ACB, and agent-wake are still partial/blocked/absent. The deploy
command works but can't be fully tested without all components installed.

## On the work done

**What went well:** Parallelizing the implementation across subagents worked
for independent modules. The adversarial review was valuable — it caught a
CRITICAL password leak in the _mask_dsn function that I would have missed. The
probe script correctly detected the new implementations and updated the feature
matrix. CI failures were caught and fixed quickly.

**What I'm confident in:** The deploy, evidence, backup, and forward-recovery
modules follow existing conventions precisely. The test coverage is solid for
the happy path and main failure paths. The _mask_dsn fix correctly handles both
URI and keyword/value DSN formats. The PGPASSWORD approach for pg_dump/pg_restore
is the PostgreSQL-recommended pattern.

**What needs a second look:** The deploy module's doctor step now imports
shared_service_components and MemoryProviderConfig inside the function — these
are deferred imports that should ideally be at module level, but the existing
codebase pattern puts them in the CLI match arm. The forward-recovery function
doesn't support dry-run and the CLI rejects the combination rather than
implementing it. The feature-probes script checks existence, not functional
correctness — a module can import and have a test file while still being broken.

## Gaps to flag

- The deploy module's lock step marks drift as DONE rather than FAILED — a
  deploy can report success with a wrong lock. This was flagged in review but
  not fixed (MEDIUM severity, needs design decision about whether to
  regenerate or fail).
- The evidence.py module has a `dsn` parameter that's never passed to any
  component CLI — it's manifest-only metadata. This is misleading.
- The backup.py evidence-export step doesn't stop the pipeline on failure —
  it records FAILED but continues to the manifest step. This is arguably
  correct (the manifest should always be written) but inconsistent with
  earlier steps.
- The feature-probes script uses `hasattr` instead of `callable()` — a
  module can expose a non-callable attribute with the right name and pass
  the probe. Low risk but worth tightening.
- The forward-recovery function's `except (KeyError, OSError)` for
  memory_provider_config may still be too narrow — if the config module
  itself is broken, ImportError would escape uncaught.
- The test_cli.py deploy stub uses a `type()` call to create an anonymous
  drift-result object — this is a code smell that should be replaced with
  a proper mock or the real DriftResult dataclass.
- The release-board JSON has all WIs with status "not_started" — this needs
  to be updated as work progresses, but there's no CI check that validates
  status transitions.
