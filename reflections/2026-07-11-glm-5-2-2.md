## On the project

The suite is at an inflection point. Phase 1's three work items (WI-1.1, WI-1.2, WI-1.3) are now implemented, but each has a "last mile" that requires live infrastructure to close:

- WI-1.1: The offline bundle verifier doesn't verify event signatures (only chain hashes and anchor roots). This is a deliberate scope decision — public key material isn't in the bundle — but it means a tampered bundle with consistent hashes but forged signatures would pass. The claims ledger documents this as a known limitation.
- WI-1.2: The live proof script (`e2e_proof.py`) has never been run against a real Claude Code session. The SQL fidelity tests close the wiring gap, but the actual exit criterion ("the live proof passes") remains a manual runbook.
- WI-1.3: 5 of 11 planned mutations require components not available at suite level (agent-wake, agent-capability-broker, live hooks). These are documented as deferred in the test file.

The claims ledger is now a genuinely useful artifact: 7 supported, 7 experimental, 0 provisional. The maturity progression tells a real story about what's proven vs. what's mechanism-only.

## On the work done

**What went well:** The delegation pattern worked effectively. Kimi and GLM subagents produced solid initial implementations, and the cross-adversarial review (GLM reviews Kimi, Kimi reviews GLM) caught real issues — including two CRITICAL findings (tautological unauthorized_project_access test, missing file write before verifier). The regista code was complex enough that implementing it directly with full context was the right call.

**What was awkward:** The regista bundle module had to be reviewed by Kimi (since GLM implemented the agent-provenance work), and the review found a CRITICAL issue (since_seq anchor slice) that required a non-trivial fix. The `since_seq` export creates a bundle that doesn't contain the full event prefix needed to recompute anchors — the fix marks anchors whose target lies before the first exported event as `skipped` rather than `failed`. This is correct but adds complexity.

**What I'm confident in:** The cross-segment chain verification is straightforward and well-tested. The conftest extraction is a clean refactor that reduces duplication. The adversarial corpus parameterization is extensible.

**What I'd want a second pair of eyes on:** The offline anchor verification logic in `_verify_anchor_offline` replicates the chain-walk from `verify_content_anchor` but operates on `Event` objects from `Event.from_dict` instead of raw DB rows. Any field encoding difference between `to_dict`/`from_dict` and the DB representation could cause silent mismatches. The tests don't cover this path with real anchor receipts + real DB events (Postgres wasn't available locally).

## Gaps to flag

- **`_bundle.py` signature verification gap** (`_bundle.py:159-262`): `verify_audit_bundle_offline` does not verify event signatures. A bundle with consistent hashes but forged signatures passes. Documented in claims ledger but not enforced.
- **`_bundle.py` since_seq anchor skip** (`_bundle.py:298-307`): Anchors whose target_global_seq precedes the first exported event are marked `skipped=True, verified=True`. This is correct but means a `since_seq` bundle can't independently verify older anchors — the auditor would need the full bundle.
- **`test_proof_wiring.py` import hack** (`test_proof_wiring.py:20-26`): Uses `importlib.util.spec_from_file_location` to import from `scripts/e2e_proof.py`. Works but is fragile — if the script moves or renames, the test breaks silently.
- **`test_adversarial_corpus.py` missing mutations** (`test_adversarial_corpus.py:42-47`): 5 of 11 WI-1.3 mutations are deferred. The `_DEFERRED_MUTATIONS` list documents them, but there's no CI gate that prevents the list from being silently deleted.
- **`test_bundle.py` untested against real DB** (`test_bundle.py`): All 13 tests skip when Postgres is unavailable. The CI interop job should run them, but this hasn't been verified yet (the interop job runs `test_interop.py`, `test_tamper.py`, `test_restore_drill.py` — not `test_bundle.py`).
- **Pre-existing `cmd_archive_verify` bug was fixed** (`_cli.py:727`): Changed `result.get("valid")` to `result.get("verified")`. This was a pre-existing bug, not introduced by this session — but it means the `regista archive verify` CLI command has been reporting "FAILED" for all segments since it was written.
