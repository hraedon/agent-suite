# Session Reflection — 2026-07-05

**Work summary:** Validated and resolved two externally-reported findings (Fable review) about the tamper-detection test (WI-2.3) and verify-restore command (Phase 4). The core fix: verify_restore.py now reads the `warnings` field from regista's replay JSON and classifies warnings>0 as WARNINGS_DETECTED (not VERIFIED), closing a false-positive where a store with a forged `prev_event_hash` was reported as "cryptographically intact." Also: added the AC-required forged-signature scenario to the tamper test, wired verify-restore into `doctor --verify-restore`, added a CI restore drill, fixed CI to run the tamper test, and resolved all HIGH/MEDIUM findings from two independent adversarial reviewers.

## On the project

agent-suite is a thin orchestration layer — it shells regista's CLI, never reimplements. The boundary is clean: `verify_restore.py` is stdlib + subprocess, `doctor.py` aggregates component health, `cli.py` dispatches. The `assert_never` discipline over closed-set enums (ProjectVerifyStatus, ComponentStatus, Command) is the family's substitute for exhaustive-match and it works — adding WARNINGS_DETECTED forced updates in `_compute_ok`, `format_text`, and the parametrized enum test, all of which the compiler/test suite caught immediately. The architecture import-boundary test (`test_architecture.py`) enforces that core modules import no backend SDK and no component code — passing `from agent_suite import verify_restore` in `doctor.py` was fine because `verify_restore` is a sibling core module, not a component.

The most striking thing: the false-positive hole was invisible from inside agent-suite. `verify_restore.py` correctly parsed what regista emitted, but regista omits `warnings` from JSON when warnings==0, and agent-suite never asked for it. The tamper test (in agent-suite's `tests/`) proved regista *detects* chain-link forgery as warnings-only, but nobody connected that fact to the verify-restore classification logic. The cross-repo review (Fable) caught it because it looked at both sides. This is the kind of gap that adversarial review across repo boundaries exists to find.

## On the work done

**What went well:** The fix is small and surgical. Adding `WARNINGS_DETECTED` to the enum, parsing `data.get("warnings", 0)`, and placing the check between drift/halted and VERIFIED — the precedence is right (drift is more severe than warnings, warnings is more severe than clean). The `assert_never` discipline meant every dispatch site was forced to handle the new status. The two adversarial reviewers (GLM and Kimi) converged on the same three HIGH findings, which gave confidence those were real.

**What was awkward:** The doctor wiring had more design tension than expected. The first attempt passed doctor's 30s runner to verify_restore, which would timeout on real stores — both reviewers caught this. The second issue: `post_restore.ok` wasn't folded into `suite_ok`, so `doctor --verify-restore --exit-code` would exit 0 on a compromised restore. The fix (set `suite_ok = False` when `post_restore.ok is False`) is obvious in hindsight but easy to miss when the field is "just attached" to the report. The `continue-on-error: true` on the regista CI install step was pre-existing but inherited by the new tests — removing it means regista install failure now fails the job, which is the right behavior for a release gate.

**What I'm confident in:** The verify_restore warnings fix, the tamper test forged-signature scenario (using `len(original_signature)` for the forged bytes), the CI changes, and the test coverage. All 128 unit tests pass, lint clean, mypy --strict clean.

**What I'd want a second pair of eyes on:** The restore drill test (`test_restore_drill.py`) hasn't run against a real Postgres — it collects and parses but I couldn't execute it (no Docker in this environment). The pg_dump/psql commands and the `verify_restore()` call against the restored DSN are untested end-to-end. The cleanup logic (outer try/finally dropping the project schema, inner try/finally dropping the restored DB) should be robust, but a real CI run is the proof.

## On what remains

**Needed before ship:**
- The restore drill and tamper tests need a real CI run to confirm they pass against the ephemeral Postgres. The CI YAML is valid and the tests collect, but execution is unverified.
- The bootstrap-contract.md §3 doctor umbrella shape doesn't document the optional `post_restore` field. Should be added for contract completeness.

**Nice to have:**
- A negative restore-drill test (corrupt the dump, assert verify_restore.ok is False) — both reviewers noted this. The unit tests cover the parser path, but the end-to-end "corrupt backup → drill fails" is unverified.
- The restore drill only dumps `--schema {project}`, not the key registry. Fine for HMAC-only, but an Ed25519 deployment with Postgres-stored keys would need the full DB dumped.
- The `format_text` for doctor doesn't clearly surface that the suite is NOT OK when only post_restore fails — it prints "post-restore verification: NOT OK" in a section, then "suite: NOT OK" at the end, which is correct but could be clearer.

## Gaps to flag

- `tests/test_restore_drill.py` — untested end-to-end (no Docker available in this session). The pg_dump/CREATE DATABASE/psql restore/verify_restore chain needs a real CI run. If pg_dump fails or CREATE DATABASE fails, the outer try/finally drops the project schema, but the restored DB cleanup is in an inner try/finally that only runs if CREATE DATABASE succeeded.
- `docs/bootstrap-contract.md:67-71` — the doctor umbrella shape doesn't include `post_restore`. The contract should document this optional field now that doctor --verify-restore exists.
- The regista replay CLI exits 0 on warnings (`_cli.py:244`) — this is a regista-side design choice (warnings are non-fatal), but it means any tool that shells `regista replay` and checks exit code will miss warnings. agent-suite's verify_restore now handles this, but other consumers (dossier, agent-notes) may have the same blind spot. Worth a cross-repo check.
- `tests/test_tamper.py` Scenario 4 uses event_seq=3 (the `submit_for_review` transition). If the canonical workflow's transition sequence changes, the seq numbers shift. Scenarios 2 and 3 both use seq=2; scenario 4 uses seq=3 — this works but is fragile against workflow changes.
- The `--restore-dsn` help says "required when --verify-restore is set" but it's not enforced — if neither `--restore-dsn` nor `REGISTA_DSN` is set, `verify_restore_dsn` is `None` and the post_restore check is silently skipped. This is safe (no crash) but confusing for the operator.
