## On the project

agent-suite is a thin orchestration layer — it composes six component CLIs (regista, dossier, agent-notes, cairn, acb, agent-wake) into a deployable unit. The design discipline is strong: stdlib-only core, `assert_never` on every closed-set dispatch, honest health reporting, no reimplementation. The AGENTS.md and bootstrap-contract.md are unusually precise — they dictate what the code *must* do before you write it. This makes implementation straightforward: you follow the contract.

The main tension I noticed: the component contracts (Plan 025 WI-3.1) aren't fully implemented yet, so running `doctor` against real components reveals rough edges — `regista doctor --json` doesn't emit `ok:`, `acb doctor --json` returns a JSON list not a dict, `agent-wake` has no `doctor` subcommand. The doctor handles these defensively (named states, not tracebacks), which is the right posture, but it means `doctor` currently reports mostly failures on this box. That will improve as components conform to the contract.

## On the work done

The core design decision was decoupling `lock.py` from `doctor.py` — lock has its own `VersionRunner`/`Installed` protocols (structurally identical to doctor's), and `doctor.py` imports `lock` (not vice versa). This prevents a circular dependency and keeps the lock module independently testable.

The most useful thing the adversarial review caught was the malformed-lock-crashes-doctor issue — both reviewers independently flagged that `deserialize_lock` raises `ValueError` and `aggregate()` didn't catch it, so a broken `SUITE.lock` would turn `doctor` into a traceback. This violates the "honest health / named state" rule. The fix (catch `ValueError` in `_check_lock_drift`, return `LockDriftResult(matches=False, note="unreadable: ...")`) was straightforward and I'm confident in it.

I'm less confident in the `lock --check` exit code change. The Nemotron reviewer argued `--check` should exit non-zero when no lock file exists (no baseline to check against). I agreed and changed it. But there's a reasonable argument that "no lock" is different from "drift" — some CI pipelines might want `lock --check` to pass on a fresh checkout before the first `lock` generation. This is a design call the user should review.

## On what remains

**Needed before this can ship:**
- WI-2.2 (suite-interop CI) — the lock is just a manifest until the interop test proves the pinned set works together. A green lock is what makes a release.
- Component doctor contract alignment — regista's `doctor --json` needs an `ok` field, acb needs to return a dict, agent-wake needs a `doctor` subcommand. These are component-side work (Plan 025 WI-3.1), not agent-suite work.
- WI-3.1 (bootstrap) — the `agent-suite bootstrap` command is still a stub. Until it exists, `agent-suite new-project` and `--user` onboarding can't work.

**Nice to have:**
- The `release` field in `SUITE.lock` is derived from `importlib.metadata.version("agent-suite")` — when the package is installed in development mode, this returns `"0.0.1"`. A real release workflow should tag and version.
- `check_drift` doesn't compare the `repo` field in component pins — a component installed from a different fork would pass if the version matches. This is a supply-chain concern flagged by the Nemotron reviewer but out of scope for WI-2.1.
- `read_regista_quad` silently returns `None` for all failure modes (not installed, crashed, bad JSON). The caller can't distinguish these. A richer return type would help debugging.

## Gaps to flag

- `format_drift_text` uses `{d.component:<22}` but `agent-capability-broker` is 25 chars — misaligned output. (`lock.py:482`)
- `agent-notes breadcrumb reconcile` failed because the `agent_suite` project schema doesn't exist in the DB yet — provisioning (WI-3.1/3.3) hasn't run. Breadcrumbs can't be filed until then.
- The `SUITE.lock` file is written to CWD (`Path("SUITE.lock")`) — in a deployed system, the lock path should be configurable from `suite.env`. Currently hardcoded.
- No test verifies `write_lock_file` is actually called by the `lock` CLI command (the CLI test stubs it to a no-op). A test that writes to a real temp path and verifies the file exists would close this gap.
- `doctor.py:_check_one` doesn't normalize `version` to `str` — a component could emit `version: 42` (JSON number) and it would be stored as `int`, then `str(version)` in `check_drift` would produce `"42"` which might not match a lock that has `"42"` as a string. In practice, all current components emit string versions, but the type contract is loose.
