## On the project

The suite is at an inflection point. The mechanisms are broad and the
architecture is sound, but the gap between "mechanism exists" and "claim is
proven" is still the limiting factor. Plan 015's gate sequence is the right
structure — it forces every assurance sentence to point to an executable proof.
Gate 0 (truth reconciliation) is mostly landed: the feature-matrix probes
work, the support matrix is ratified, the release board exists. What's missing
is the inventory CLI (WI-0.2 proof command doesn't exist yet) and the actual
Gate 1-5 execution.

The support matrix dataclass in `release_artifacts.py` was originally a
subset of the JSON file — the adversarial review caught that `to_dict()`
silently dropped 7+ fields. This is exactly the kind of claim-vs-evidence gap
the holistic review (F-1 through F-6) was about: the proof command validated
the Python default, not the canonical artifact. The roundtrip tests now guard
against this class of drift.

## On the work done

The changes were small but high-signal:
- The adversarial corpus registry guard is now bidirectional and tracks all 4
  standalone tests (not just 2). This prevents `replayed_wake_event` and
  `capability_clobber` from silently vanishing.
- The support matrix now models all fields in the JSON file, with roundtrip
  tests that load the canonical artifact and verify no fields are lost.
- CI now targets PostgreSQL 18 and Python 3.14 (including a Windows matrix).
- Claims ledger CL-010 is more honest: it acknowledges the revoked_key test
  exists while being clear it exercises `verify_event_principal_binding`, not
  the full `replay` chain.

What I'm confident in: the roundtrip tests, the registry guard, the CI updates.
What I'd want a second pair of eyes on: whether `postgres:18` Docker image
works in GitHub Actions (it's a valid tag on Docker Hub but untested in CI
yet), and whether Python 3.14 is available on `windows-latest` runners.
