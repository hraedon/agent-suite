## On the project

The agent-suite is maturing rapidly. Plan 014 turns the protocol foundations
from Plan 013 (which defined types and fail-closed protocols) into running
code with real executors, state stores, and JWKS integration. The codebase
now has a coherent arc: preflight → plan → apply_plan → signed receipt, and
dual control with a real state store that prevents replay.

The architecture is holding up well under growth — the `assert_never` pattern,
frozen dataclasses, and stdlib-only core with lazy edge imports are all
consistent. The architecture test caught a real boundary violation
(config.py importing from entra.py at module level) that was easy to fix.

What feels fragile: the CLI dual-control commands use hardcoded principal
IDs ("cli-requester", "cli-approver") instead of real token validation.
This is acknowledged as a skeleton — the Entra adapter exists but isn't
wired into the CLI. That's the right sequencing (build the protocol first,
wire real auth later), but it means the CLI is currently a security facade.

## On the work done

**What went well:** Parallelizing implementation across 4 subagents (2 for
initial implementation, 2 for adversarial review, 2 for fixes) was highly
effective. Each phase was independent enough to avoid file conflicts. The
adversarial review was the most valuable step — it caught a critical dual
control bypass (H-1: evaluate_approval using caller-supplied request instead
of stored request) that would have defeated the entire security model.

**What I'm confident in:** The dual control state store is correct after
the fixes — stored request is used for all checks, expired approvals can't
be executed, file locking handles Windows fd leaks. The SigningKeyStore
is race-safe with O_CREAT|O_EXCL and 0o600 permissions. The Entra tid
validation now fails closed.

**What needs a second look:** The JWKS caching change (from caching a single
key to caching the PyJWKClient itself) is correct in principle but the tests
use mocks that may not perfectly mirror PyJWKClient's real behavior. The
TokenCache is not thread-safe (consistent with the rest of the codebase,
but worth noting). The PowerShell installer script is syntactically valid
but has never been run on actual Windows.

## What remains

- **Wire Entra validation into the CLI** (HIGH priority): The dual-control
  CLI commands construct ValidatedToken with hardcoded principals. Need to
  call EntraTokenValidator.validate() when Entra is configured, and fail
  closed when it's not.
- **Real Windows testing** (HIGH priority): The Windows setup executor,
  WinSW install, DPAPI key store, and PowerShell installer have never run
  on actual Windows. The tests use mocks and custom runners.
- **WinSW binary checksum** (MEDIUM): The install-winsw.py script has a
  placeholder SHA-256 that must be verified against the official release.
- **Token on command line** (MEDIUM): The CLI passes --token as a visible
  process argument. Should read from env var or stdin.
- **mark_executed fallback operation** (LOW): When store.get returns None,
  mark_executed falls back to SERVICE_REPAIR as the operation, which is
  misleading.
- **Plan 014 status update**: The plan is marked "Proposed" but all 4 phases
  are now implemented. Should be updated to "In Progress" or "Landed".

## Gaps to flag

- **CLI dual-control is a security facade** (`cli.py:644-650, 688-694`):
  Hardcoded principals "cli-requester"/"cli-approver" bypass all real
  authentication. Same person can be both requester and approver.
- **_default_runner for REPAIR is a no-op stub** (`windows_setup.py:547-548`):
  Returns APPLIED without doing anything. Misleading for a deployment tool.
- **APPLY_SIGNED_BUNDLE always fails** (`windows_setup.py:545-546`):
  Returns ActionState.FAILED unconditionally. Not yet implemented.
- **TokenCache not thread-safe** (`entra.py:55-89`): Uses OrderedDict without
  locking. Fine for CLI use, problematic for web server use.
- **DualControlStore temp file is predictable** (`dual_control_store.py:224`):
  `path + ".tmp"` is a predictable name. Symlink attack vector if the
  directory is writable by other users.
- **No tests for _default_runner subprocess calls** — all tests use custom
  runners. The WIRE_HARNESSES command fix (adding project slug) is untested
  against the real CLI.
- **Entra JWKS tests mock PyJWKClient** — no integration test against a real
  JWKS endpoint. The mock may not perfectly mirror real behavior.
- **install-winsw.py has placeholder checksum** (`scripts/install-winsw.py:24`):
  `EXPECTED_SHA256 = "sha256:PLACEHOLDER_VERIFY_BEFORE_RELEASE"`. Must be
  filled in before release.
