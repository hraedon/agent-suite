## On the project

This is the closest the suite has been to a single deployable system in the time I've worked on it. The pieces finally compose: regista's public secret facade is published; the four faces and plumbing tiers each ship a component-owned Codex plugin bundle that the suite marketplace composes; the suite umbrella's doctor can see every component's real check detail (no more "exit 1: no stderr"); the lock pins what's actually installed, including the SHAs.

What still feels fragile: the suite is held together by **pip-install-git-pin** conventions that aren't enforced anywhere. agent-provenance's CI broke the moment a regista pin tightened to `>=0.5.1,<0.6` without a PyPI release; agent-notes and dossier avoid this only by pre-installing regista from git in a CI step. If the family is serious about "reproducible candidate definitions," regista needs an actual PyPI release (or a private index) — the SUITE.lock revision SHAs are only as immutable as the install path that consumes them, and right now a wheel install silently degrades to version-only.

The adversarial-review discipline caught real things this session. The `inventory.py` module shipped a docstring that lied about `assert_never` coverage. The lock accepted `revision = ""` and then falsely reported drift. The link-audit fix smoothed orphaned soft-deleted memories into "cross-project reference" — exactly the kind of dishonest health the AGENTS.md "honest health" rule exists to prevent. None of these were caught by the test suite; they were caught by a skeptical second pass. The family convention of "implementer-provided adversarial review when dedicated reviewers are usage-locked" is doing real work and should be preserved.

## Gaps to flag

- **regista 0.5.1 is not on PyPI** (`/projects/regista` is at 0.5.1 in pyproject.toml; `pip show regista` reports 0.0.0 from PyPI). Every component CI that pre-installs from `git@main` is depending on the tip of main not breaking them. This is the single biggest reproducibility gap.
- **`_suite_release()` reads `data/release-board.json` via `Path(__file__).resolve().parents[2]`** (`src/agent_suite/lock.py`). This is fragile for wheel installs — the data dir isn't included in the wheel build today. The `inventory.py` `_default_inventory_path()` has the same issue. M-2 from the adversarial review was deferred.
- **`test_drift_kind_format_is_total` is parametrized over `list(DriftKind)`** but the new `REVISION_MISMATCH` was added at the same time as the format case — so the test didn't catch a missing-format case because there wasn't one. Future drift kinds need the format case landed *first* to make the test do its job.
- **The link-audit fix's strict set is negatively enumerated** (`<> 'relates_to'`). The docstring now describes this honestly, but if a future soft relationship is added (e.g. `see_also`), it will be treated as strict by default and fail the audit. The doc at `src/agent_notes/scripts/doctor.py:134` calls this out — the next maintainer should know.
- **The convergence proof did NOT exercise agent-wake** (no Node on mvmhermes01) and did NOT exercise Codex plugins end-to-end (no `codex` CLI). Both are documented honestly in `golden/convergence-proof-20260718T232221Z.md` but the proof's headline "CONVERGED" should not be read as "every tier verified."
- **The dogfood suite.env on this machine uses real hostnames** (`mvmpostgres01.ad.hraedon.com`, `hindsight-api.k8s.hraedon.com`) — those are in `~/.config/agent-suite/suite.env` (correctly gitignored), not in any committed file, but the proof log at `golden/convergence-proof-20260718T232221Z.md` was scrubbed of real identifiers. Worth a final identifier-gate run before any public flip.
