## On the project

The suite is architecturally sound — thin orchestration over six components, each with a clear role. The blueprint and bootstrap-contract docs are genuinely good design documents that I didn't need to second-guess. But the gap between "designed" and "deployed" is where the real work lives, and this session was entirely in that gap. The suite had never been exercised on a real box. Every component had been tested in isolation (unit tests, CI), but the composition — the actual bootstrap calling actual CLIs — had never run end-to-end. That's where 7 of the 8 bugs lived: the bootstrap assumed `regista secrets probe` existed (it doesn't), assumed `install-harness --harness all` was the argument shape (it's positional), and the doctor umbrella assumed every component emits a top-level `ok` field (regista doesn't). These are integration bugs that no amount of unit testing in isolation would catch — they require the actual composition.

The packaging issues were the most instructive. cairn's `_hermes_source_plugin_dir` goes 3 levels up from `_install.py` to find the repo root, then looks for `integrations/hermes/`. In the dev repo this works. In an installed wheel, 3 levels up lands at `python3.14/` — wrong place. Same pattern in agent-notes' `_repo_skills_root` (4 levels up). The fix was to look relative to the package, not the repo root, and include the directories in the wheel via `force-include`. This is a classic "works in dev, breaks on install" pattern that every Python package with non-Python data files hits.

## On the work done

I'm confident in the code fixes. The 8 fixes are small, targeted, and each has a clear before/after. The suite.env auto-loader (`config.py`) is the biggest simplification — it means `agent-suite doctor` just works without manual `source suite.env`. The deployment guide is thorough (12 sections, covers onboarding new collaborators, troubleshooting, quick reference).

What I'm less confident about: I didn't write new tests for all the fixes. The `config.py` suite.env loader has no dedicated test (the existing CLI test was patched to isolate from the host's suite.env, but there's no test that verifies the loader actually injects vars). The regista `DoctorReport.to_dict()` change has no new test (the existing doctor tests need a real Postgres and fail without one). The packaging fixes (`force-include` in pyproject.toml) have no test — I verified them manually by reinstalling on WORK-STATION-01. These should be backfilled.

The deployment to WORK-STATION-01 is real and working. The doctor reports regista OK, cairn OK, agent-notes 1 pre-existing data fail (11 dangling links in the shared store). The Hermes .env has the cairn managed block. The cairn plugin is installed at `~/.hermes/plugins/observability/cairn/`. SUITE.lock is generated and matches. The only thing I didn't do is start Hermes — the service is inactive, so no actual tool-call attestations have been produced yet. That's the operator's step.

## Gaps to flag

- **No test for `config.py` suite.env loader** — `src/agent_suite/config.py` is new and only exercised via the patched CLI test. Needs a dedicated test that verifies vars are injected into `os.environ` and that process env wins over suite.env.
- **regista doctor tests require a live Postgres** — `tests/test_doctor.py` in regista has 2 tests that fail without a local Postgres (`test_reachable_db_with_no_projects`, `test_with_project_checks_schema`). These are pre-existing integration tests, not caused by my changes, but they mean the `ok` field fix is unverified by CI.
- **Packaging fixes unverified by CI** — the `force-include` rules in cairn and agent-notes `pyproject.toml` were verified manually on WORK-STATION-01 but have no automated test. A wheel-build test in CI would catch regressions.
- **SUITE.lock path is relative** — `DEFAULT_LOCK_PATH = Path("SUITE.lock")` in `lock.py:30` is CWD-relative. The doctor only finds the lock if run from the directory where `agent-suite lock` was invoked. This should be an absolute path (e.g. `~/.config/agent-suite/SUITE.lock`) or the doctor should search multiple locations.
- **agent-notes `resolve_dsn` fallback is partially redundant** — I added the suite.env fallback, but agent-notes' `RegistaConfig` already reads `REGISTA_DSN` from suite.env. The native DB path (`resolve_dsn`) and the regista face path (`RegistaConfig`) use different resolution code. They should converge — the native DB DSN and the regista DSN are often the same host.
- **The `suite-service` principal created by bootstrap is orphaned** — the bootstrap provisioned `suite-service` on WORK-STATION-01, but the actual principal is `WORK-PRINCIPAL`. The `suite-service` principal has a key in the store but nobody uses it. WI-1 fixes the root cause; the orphaned principal should be cleaned up.
- **Hermes harness version is "Hermes"** — the cairn managed env block sets `CAIRN_HARNESS_VERSION=Hermes` (the literal string), not an actual version. `cairn install-harness hermes` should detect the Hermes version (e.g. from `hermes --version`).
