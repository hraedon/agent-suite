# Session Reflection — 2026-07-12

**Work summary:** Implemented Plans 012 (agent-suite) and 020 (agent-notes) —
the pluggable memory engine composition. This makes agent-notes a seam between
exact signed knowledge (regista) and learned memory (Hindsight), with native
pgvector as the always-available fallback. Both plans went through adversarial
review; all critical/high issues were fixed before push. All four agent targets
(opencode, claude code, codex, hermes) are wired to use Hindsight as the
learned-memory backend via MCP and suite.env. CI green on both repos.

---

## On the project

The agent-suite/agent-notes split is the right architecture for this. agent-notes
owns the provider protocol, adapter, and CLI; agent-suite only carries config,
delegates via subprocess, aggregates health, and pins the provider in the lock.
The thin-orchestration boundary held cleanly — agent-suite never imports
agent-notes code, it shells `agent-notes memory-provider <verb> --json`.

The Hindsight API is well-designed for this use case. One bank per project scope,
async operations with status polling, document-level deletion with cascade, and
the MCP endpoint at `/mcp` makes agent wiring trivial. The live instance at
hindsight-api.k8s.hraedon.com is open (no auth), which simplifies deployment
but should be noted for the security posture.

## On the work done

The protocol definition (`memory_engine.py`) is the strongest piece — typed
frozen dataclasses for every request/result, enums for capabilities/states/origins
with `assert_never` in every dispatch, and a factory that reads through the full
precedence chain (env > suite.env > config file > default).

The adversarial review caught real bugs: `assert_never` misuse on external API
strings (would have crashed on any unknown Hindsight status/type), URL path
injection in source_id/operation_id, unhandled int/float conversions on external
responses, uncaught DB exceptions in native engine, secret exposure in exception
messages, and the memory_types filter being a no-op. All fixed.

The agent wiring was straightforward once the suite.env integration was fixed.
The initial gap: `get_engine()` and `HindsightEngine.from_config()` didn't read
suite.env — only env vars and config file. Fixed by adding suite.env as a
precedence layer between env and config file, matching how `RegistaConfig`
already works.

## Gaps to flag

- `read_provider_extension` in `lock.py` hardcodes `deployment_mode="remote"`
  and `support_level="supported"` — these should come from agent-notes'
  describe output once the contract includes them
- `OperationState.STALE` is defined but never produced by either engine
- Hindsight `describe()` doesn't inspect the health response body — if
  `/health` returns `200` with `{"status":"unhealthy"}`, the adapter reports
  HEALTHY. The live instance returns plain `200` with `{"status":"ok"}` so
  this is not currently a problem
- The contract's `support_levels` list `["supported", "experimental"]` but
  Plan 012 WI-0.2 defines four levels (`experimental`, `qualified`,
  `recommended`, `unsupported`) — the contract should be updated
- No config validation for contradictory settings (engine=hindsight with no
  URL, engine=native with HINDSIGHT_URL set) — the bootstrap catches the
  former but the config function itself doesn't reject or warn
- Hermes agent-notes was synced via rsync — the hermes repo is not a git
  remote, so future changes need manual sync or a proper deployment pipeline
- The `export` CLI command specified in Plan 020 WI-3.1 is not implemented
