## On the project

agent-suite is a thin orchestration layer that is growing into its role as the release authority. The Plan 015 gate structure is sound — it converts the feature and robustness roadmaps into evidence-producing checkpoints. The key tension is that the layer must be honest about what it has proven vs. what it merely implements, and this session was primarily about closing that gap.

The codebase has strong conventions (assert_never, closed-set enums, stdlib-only core, frozen dataclasses) that make it easy to navigate and hard to silently break. The release_artifacts.py module is a good example — it has both Python defaults and JSON file parsing, but the proof commands were validating the defaults rather than the actual JSON. The adversarial review caught this. The fix (updating defaults to match JSON) is correct but fragile — a future agent could re-diverge them. The real fix is for the proof commands to validate the JSON files directly, not the defaults.

The identifier gate replacement was the most satisfying fix. The old hand-rolled gate had `hraedon` (the published author identity) in its denylist — exactly the inversion the publication-prep skill warns about. The canonical gate uses only the work-domain (El Rio) denylist and scans only tracked files. This is both more correct and more maintainable.
