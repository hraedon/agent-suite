## On the work done

**What went well:** The revoked_key mutation was clean — generating an Ed25519 keypair, registering the public key, creating an in-memory Ed25519-signed event, and verifying that verify_event_principal_binding correctly reports "key-revoked" after revocation. The in-memory approach avoids DB insertion complexity while testing the right detection path. The hook_omission test was similarly straightforward — cairn's _check_harness_wired function is well-factored and the CAIRN_CLAUDE_SETTINGS env var makes it trivially testable.

**What was awkward:** The corrupted_backup test took three attempts. The first approach (text replacement in pg_dump --inserts output) failed because psql doesn't stop on error by default, so broken INSERTs left only 1 event restored, and that 1-event chain was internally consistent. The second approach (restore clean dump then corrupt via SQL UPDATE) was correct but exposed a pre-existing bug in verify_restore: it treated regista replay's exit code 1 (drift detected) as UNREACHABLE instead of parsing the JSON output. The fix was simple but the bug had been hiding since verify_restore was written — the existing test_restore_drill only tests the positive case (clean restore passes), never the negative case (corrupted restore fails).

**What I'm confident in:** The revoked_key and hook_omission tests are well-structured and test the right detection paths. The verify_restore fix (parse JSON before checking exit code) is obviously correct — the JSON output is authoritative, and the exit code is just a convenience signal for shell scripts.

**What I'd want a second pair of eyes on:** The corrupted_backup test's SQL UPDATE uses `jsonb_set(payload, '{custom_fields,title}', '"CORRUPTED-BACKUP-TEST"'::jsonb)`. This assumes the payload has a `custom_fields.title` path, which is true for the `_drive_to_done` setup (it sets `custom_fields={"title": "Adversarial corpus work-item"}`). If the payload structure changes, the UPDATE might not match and the test would fail in a confusing way.

The existing test_restore_drill only tests the positive case — there was never a negative test to catch this. Consider adding a permanent negative test to test_restore_drill.py.
- **corrupted_backup test payload path assumption** (`test_adversarial_corpus.py:850`): The SQL UPDATE assumes `payload->custom_fields->title` exists. If _drive_to_done changes its payload structure, the UPDATE silently affects 0 rows and the test fails confusingly.
