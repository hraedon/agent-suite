## On the project

Plan 008 is a genuinely well-structured integration roadmap. The phase sequencing (contract → truth → product → deploy → operate → security) is the right order: you can't prove deployment safety on top of unproven integrity claims. The claims ledger is the most valuable artifact — it forces every assurance statement to point to an executable proof and an adversarial proof, and it made it trivially clear which findings were blocking (provisional maturity) and which were already supported.

The thin-orchestration principle is holding. All F-1/F-2/F-3 fixes were in the constituent repos (regista, agent-provenance), not in agent-suite. The orchestration layer only updated documentation (claims ledger, plan status). This is the right boundary.

The gap between "implemented" and "committed" was the main finding. The prior session's reflection noted that the anchoring and live-proof work was uncommitted. This session was essentially: review the uncommitted work, fix issues, commit, and verify. The fact that substantial F-1/F-2 fixes sat uncommitted for a day is a process risk — uncommitted work is invisible to CI, adversarial review, and other agents.

## On the work done

**Confident in:** The regista anchoring fix is sound after the adversarial review pass. The kimi subagent found three real High issues (ambiguous binding serialization allowing cross-project collisions, hardcoded SHA-256 ignoring event hash_alg, failure_count reset on re-submission) that I fixed before the final commit. The binding collision test is the kind of adversarial test that should have been there from the start — `compute_content_anchor("p1", 2, 3)` vs `compute_content_anchor("p", 1, 23)` producing the same hash was a genuine cryptographic weakness in the original implementation.

**Less confident in:** The FileAnchorProvider.verify fix (matching on merkle_root only instead of merkle_root + receipt_id) is correct but exposes a design tension: the AnchorProvider protocol's `submit(merkle_root)` doesn't receive a receipt_id, so the provider generates its own, which then gets overwritten by trigger_anchoring. The file log has the submit-generated UUID but the DB has the trigger-generated UUID. Matching on merkle_root alone works because it's unique per batch, but the protocol could be cleaner.

**The spec.md fix was important.** The original spec text claimed the anchor "reuses Plan 012's `compute_merkle_root`" and "computes a Merkle root over a batch of recent events" — but the F-1 fix completely replaced this with content anchoring. The spec was actively misleading. I also corrected the §17.9.2 verification description, which understated what `verify_anchor_receipt` does (it claimed verify doesn't recompute from live events, but it does). Spec accuracy matters more than code accuracy for the claims ledger — a reviewer trusts the spec over the code.

**The agent-provenance CI failure** (`test_install_claude_detects_version_when_unknown`) was a classic "works locally, fails in CI" issue: the installer was writing `CAIRN_HARNESS_VERSION=unknown` to the settings file when Claude Code wasn't installed. The fix (drop the key when detection fails instead of leaving "unknown") is cleaner than the original behavior.

## Gaps to flag

- **`actor_kind`/`actor_metadata` not in canonical envelope** — `src/regista/_signing.py:97-136` (`build_signing_envelope_v4`). The spec says these are signed; the code doesn't. This means an operator with DB access can mutate an agent's role/model/provider metadata without detection by the anchor, replay, or signature verification. Documented in CL-012 residual risk but not fixed.
- **`committed` status is dead code** — `src/regista/_anchoring.py:36` declares `AnchorStatus.COMMITTED = "committed"` but no provider ever returns it and no code path sets it. Either remove it or define a provider that uses it.
- **No test for non-SHA-256 events with anchoring** — The fix to use `resolve_hash_function` in `verify_content_anchor` is correct but untested with actual `hash_alg="sha-384"` events. The test suite only creates SHA-256 events. A test with a different hash algorithm would verify the fix.
- **Stuck pending receipts after crash** — If `provider.submit()` succeeds but the DB update fails (crash between steps 2 and 3b), a `pending` receipt with `receipt_bytes=None` is left in the DB. For file/RFC3161 providers, no maintenance path recovers it. `retry_failed_anchors` only processes `retryable` status. This is a Medium finding from the adversarial review.
- **`latest_confirmed_seq` includes `retryable`** — `src/regista/_anchoring.py:585-590` returns the max `target_global_seq` across pending/committed/confirmed/retryable. A permanently `retryable` receipt will be treated as the watermark, blocking re-anchoring of those events. Should exclude `retryable` from the watermark.
- **agent-provenance live proof not tested in production** — All 121 tests use constructed `ProofEvent` rows. No test runs the full `e2e_proof.py` script against a live Claude Code session with a real Postgres store. The script itself is untested in CI.
- **regista full test suite hangs locally** — The full `pytest tests/` run hangs (likely on tests that need external services or have long timeouts). Only the anchoring tests (78) pass cleanly. CI runs the full suite but local verification is limited. The test suite should mark integration tests that need external services so they skip cleanly.
