## On the project

The suite is genuinely further along than the blueprint's phase table
suggests — most "Proposed" plans turned out to be 60-90% design-complete
docs waiting for focused implementation, not open research questions. The
strongest thing about this codebase family is the consistency of its
contracts: the same ideas (render-the-record-don't-re-derive, no-raw-key-
in-the-UI, inject-don't-surface, doctor `ok`/`degraded` shape, assert_never
exhaustive dispatch, dry-run-non-acting) recur across all six components,
and an implementer who internalizes one repo's AGENTS.md can mostly predict
the others. That consistency is what let a single session land work in seven
repos without cross-pollination bugs.

The weakness is the gap between "implemented" and "proven against reality."
A large fraction of the remaining WIs across the suite are explicitly
environment-gated (live Postgres, Windows host, Vault, LDAP/Entra, a real
Claude Code session, the public flip). The code is CI-green against fixtures
and in-memory backends, but the differentiating claims — "a human can verify
what the agents did," "the chain is provably unaltered," "key custody never
touches disk" — are only proven against synthetic inputs. That's not a
defect of this session; it's the honest next gate.

## On the work done

The adversarial-review-then-merge loop earned its keep repeatedly. The
reviews caught real issues the implementing subagents (and I) missed: a
**critical SSRF** in agent-wake's webhook URL validation (only checked the
scheme, not the resolved IP), a **medium custody bug** in dossier 015
(passing a backend-ref `hmac_key_path` to a function expecting a filesystem
path, writing a bogus file and leaking `secret_ref`), and a **critical
contract gap** in agent-notes 018 (the note path *claimed* outbox/reconcile
parity with breadcrumbs but didn't implement it). None of those would have
surfaced from green tests alone — the tests tested the happy path the code
assumed. Three different reviewer lineages (kimi, glm, minimax, nemotron)
each caught distinct things, which validates the cross-lineage discipline
even when review is "same-model" in the deployment sense.

What I'm confident in: the crypto work (regista 030 reuses the signing
envelope correctly — encryption doesn't break chain/signature verification,
verified by tamper tests), the capture-correctness fix (provenance 009's
reproducible-digest discipline), the doctor conformance (acb 006 mirrors
cairn exactly), and the Hermes merges (the two fixes — SSRF and session_id —
were concrete and well-tested).

What I'd want a second pair of eyes on: dossier 015's break-glass now
*surfaces* revoke failures rather than aborting, which is a behavior change
an operator should sanity-check; agent-provenance 010's content capture is
behind opt-in but the stance-vs-reality mismatch (attests "encrypted" when
no key is configured, storing plaintext) is documented-not-fixed; and
agent-notes 018's `list_note_entities` still does a client-side scan with a
hard 10k cap because regista's read API has no `entity_kind` filter — that's
a real scaling cliff flagged honestly but not closed.

Process note: the long-running subagents (full-suite pytest via
testcontainers) hit two timeouts/aborts mid-session. Restarting fresh was
always cleaner than salvaging partial work — the partial diffs from stuck
agents weren't worth the risk of inheriting a half-baked design.
