# Session Reflection — 2026-07-12 (Session 3)

**Work summary:** Implemented three major feature areas for Plan 013 (Full v1
Human Surfaces): Windows observation adapters + preflight CLI (Area 6),
signed execution receipts + WinSW + DPAPI helper (Area 7), and dual control
protocol + Entra step-up adapter (Area 8). Two rounds of adversarial review

## On the project

The agent-suite orchestration layer is reaching a turning point. The
protocol-first design (windows_setup.py skeleton, then windows_observation.py
adapters, then CLI) is working well — each layer builds on the one below
without reimplementing logic. The architecture test (import boundary) is
load-bearing: it caught the temptation to put pywin32 imports in core modules
and forced the edge-module pattern.

The dual control protocol is the most security-critical code in this repo.
The fail-closed design (every check failure → REJECTED, never APPROVED) is
correct, but the protocol is stateless — replay protection is the caller's
responsibility. This is documented but not enforced. The Entra edge adapter
now requires a signing key (fail-closed), which is correct but means the
adapter can't be used in development without a JWKS key. That's the right
tradeoff.

## On the work done

**What went well:** The three areas were mostly independent in file creation
(windows_observation.py, winsw.py, dpapi.py, dual_control.py, entra.py), so
they could be developed in parallel. The adversarial review was extremely
valuable — it found real security issues (key_ref forgery in signed receipts,
ACR fail-open in Entra, same-principal bypass via direct DualControlApproval
construction, step-up downgrade) that the unit tests didn't catch.

**What I'm confident in:** The dual control protocol's fail-closed behavior
is thoroughly tested (14 rejection paths). The signed receipt canonical form
now includes key_ref, preventing key_id forgery. The Windows observation
probes never raise (broadened to catch Exception). The DPAPI helper correctly
detects availability and wraps all errors.

**What needs a second look:** The Entra adapter's JWT decode uses a mock in
tests (`_MockJWT` that skips signature verification). In production, the
`signing_key` parameter must be a real JWKS key. The adapter doesn't fetch
JWKS automatically — the caller must provide the key. This is by design (the
caller may want to cache keys), but it means the adapter is not turnkey.

The WinSW `install_winsw_service` now calls `winsw.exe install` via the
runner, but we couldn't test this on the Windows VM because `winsw.exe` is
not installed there. The runner is injectable, so the code path is tested
with a mock, but the actual `winsw.exe` invocation is untested on a real
Windows host.

The `probe_service_account` heuristic is weak (checks SESSIONNAME env var
and USERNAME prefix). It returns UNKNOWN in most cases, which is the right
fail-safe behavior but not very useful. A proper check would use
`OpenProcessToken` + `GetTokenInformation`, but that's a deeper Windows API
integration.

## What remains

- **Plan 013 Phase 1 (Daily team product):** The Windows setup protocol
  is now implemented but not wired into a real installer or UI. The preflight
  CLI exists but the install/apply/repair operations are still protocol-only
  (dry_run + no_op + blocked states). A real executor that actually installs
  artifacts, configures WinSW services, and wires harnesses is the next step.

- **DPAPI on Windows:** The DPAPI helper works but fails on the test VM
  because the SSH session doesn't load the user profile (DPAPI master key
  unavailable). This is a known Windows limitation — the fix is to run
  the DPAPI operations in a context with a loaded user profile (scheduled
  task, Windows Service, or interactive logon). Document this in the
  Windows install guide.

- **Entra JWKS integration:** The Entra adapter requires a signing key but
  doesn't fetch JWKS from Microsoft's discovery endpoint. A production
  implementation would use `jwt.PyJWKClient` to fetch and cache keys.

- **Dual control state management:** The protocol is stateless. A state
  store (file-based or DB-backed) is needed to track PENDING → APPROVED →
  EXECUTED transitions and prevent replay. This is the caller's
  responsibility but should be documented or provided as a utility.

- **WinSW binary distribution:** The WinSW XML generator works, but the
  actual `winsw.exe` binary is an operator prerequisite. A download script
  or package dependency would make this turnkey.

## Gaps to flag

- `entra.py:78` — The Entra adapter validates `tid` (tenant_id) against
  config, but only when `tid` is present in the token. A forged token
  without `tid` would pass the tenant check. This is mitigated by the
  signature verification (which requires the correct `aud` and `iss`),
  but defense-in-depth would require `tid` unconditionally.

- `dual_control.py:194-280` — `evaluate_approval` is stateless. The same
  (request, approval, approver_token) triple can be evaluated repeatedly,
  always returning APPROVED. Replay protection is the caller's
  responsibility. This should be documented in the module docstring.

- `windows_observation.py:59-69` — `probe_service_account` uses a weak
  heuristic (SESSIONNAME env var, USERNAME prefix). It returns UNKNOWN in
  most real scenarios. A proper check would use Windows Token APIs.

- `winsw.py:126-170` — `install_winsw_service` calls `winsw.exe install`
  via the runner, but the actual `winsw.exe` invocation is untested on a
  real Windows host. The mock runner verifies the command is called, but
  not that `winsw.exe` accepts the command.

- `dpapi.py:38` — `protect()` calls `CryptProtectData` which fails in SSH
  sessions on Windows because the user profile (and DPAPI master key) is
  not loaded. This is a Windows limitation, not a code bug, but it should
  be documented in the Windows install guide.

- `cli.py:445-485` — The preflight CLI handler uses `frozenset(SetupOperation)`
  which requests ALL operations. This means elevation and service_account
  are always required. A profile-aware operation set (e.g., only
  WIRE_HARNESSES for Profile A) would be more useful.

- `test_entra.py` — Tests use `_MockJWT` that skips signature verification.
  The real `jwt.decode` with `algorithms=["RS256"]` and a signing key is
  untested. A test with a real RS256 key pair would close this gap.
