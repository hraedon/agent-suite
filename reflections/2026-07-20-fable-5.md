## On the project

The suite crossed a threshold this session: the CLI contract stopped being
a convention and became an enforced artifact. What strikes me is how fast
the conformance kit paid for itself — the *first dogfood run* against
agent-suite's own CLI found three real defects (two traceback paths in
dual-control, one error printed to stdout), and while writing this
session's own tooling I hit the exact defect classes Plan 018 catalogues,
live, in `agent-notes work-item file --json`. The evidence base for this
plan is unusually honest because the operators writing it are also its
victims.

An inherited observation worth preserving: I found the entire Gate 0
recertification scope sitting *uncommitted* in the agent-suite working
tree — good work (it survived my review nearly unchanged) with no commit,
no reflection, no handoff. If the session that produced it had crashed
differently, it would have been lost or, worse, half-lost. The /end
discipline exists for exactly this.

## On the work done

**Confident in:** the Gate 0 tree (validated then hardened by CI — the
strict feature-probes job passed on its first real run, which means the
lock revisions, sibling installs, and probe matrix all genuinely agree);
the regista stderr default (regression-tested from both directions:
library-default-to-stderr and app-config-wins); the agent-notes
conversion (the 27-failure scare was my own test poisoning structlog
global state — reduced to the 2 pre-existing operator-box failures plus
one legit shape assertion).

**Would want second eyes on:** the retryable-code set in regista
(`CLAIM_CONTESTED`, `CONCURRENT_MODIFICATION` only — deliberately
conservative, but someone who knows the store's failure modes may want
`DB_NOT_FOUND` or timeout classes marked); and the decision to keep
agent-notes' 2/3/4 exit codes (contract says 1/2 — I chose compatibility
plus envelope-carries-semantics over a breaking renumber; that trade is
defensible but unratified).

**Mechanical transform caveat:** the 44-site agent-notes conversion was
regex-driven with a hand-reviewed mapping. The import-fixup regex broke
multi-line imports on the first pass (caught by ruff/syntax immediately).
Codes assigned by pattern (`VALIDATION_FAILED`, `NOT_FOUND`) are coarser
than hand-chosen ones would be; refining them is cheap later because
codes-added is non-breaking, codes-renamed is breaking.

## Gaps to flag

- **`breadcrumb reconcile` produces false resolution suggestions** when
  commit subjects use plan-local WI numbering (regista item 030 matched a
  "Plan 030" commit; agent-suite GJ WI-012/014 matched Plan 015's internal
  numbering). Filed as agent-notes WI-027. Until fixed, do not `--apply`
  without reading titles — six of six suggestions this session were false.

  the P0 work merged after the 0.5.2 tag. Library consumers get the stderr
  default only from git installs until 0.5.3 is published; agent-notes is
  protected regardless (it configures structlog itself).
- **agent-notes still has ad-hoc error shapes** in export-index refusals
  and memory_provider (all exit nonzero; filed WI-028).
- **Operator-box-only test failures** in `test_memory_provider_cli`
  (hindsight env-dependent; filed WI-029). Anyone running the full suite
  locally should expect exactly those two failures.
- **The conformance kit's secret-redaction fixture is shallow** — it
  plants a sentinel in env and asserts non-appearance; it does not prove
  redaction of secrets embedded in config files or DSNs. Deepen when a
  component actually handles secrets in error paths (acb is the one to
  watch).
- **GitHub API 503s** repeatedly killed `gh run watch` this session;
  the retry-loop pattern in scratchpad polling worked. If CI results seem
  missing next session, it may be the API, not the workflow.
