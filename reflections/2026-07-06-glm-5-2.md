## On the project

agent-suite continues to be what it claims to be — a thin orchestration layer
that composes components via their CLIs and never reimplements them. The
architecture is clean: `cli.py` dispatches, `doctor.py` aggregates, `verify_restore.py`
is stdlib + subprocess, `bootstrap.py` orchestrates. The `assert_never` discipline
over closed-set enums (Command, ComponentStatus, ProjectVerifyStatus) is the
family's substitute for exhaustive-match and it works — adding a new status or
command forces every dispatch site to handle it.

The most interesting design tension in this session was about *skip vs. fail
semantics in CI*. The existing interop test had a clean local-dev skip guard
(module-level `pytestmark`), but that same guard was the thing that made the
face-level proof "green-by-skip" in CI — the exact bug Plan 002 was created to
fix. The fix required making the skip logic *per-test* rather than module-level,
so the face-level test's "require" logic is independent of the spine-level test's
prerequisites. This is a generalizable lesson: a skip guard that's convenient for
local development is a liability in CI, and the two need different gates.

## On the work done

**What went well:** The core change was small and high-leverage — removing
`continue-on-error` from the CI face-install step, adding `INTEROP_REQUIRE_FACES=1`,
and wiring the fail guard. The plan was right that "the code under test already
works; the gap is purely that CI wasn't enforcing it." The CI run confirmed all
4 interop tests PASSED (not skipped) on the first push.

**What the adversarial review caught (and I'm glad it did):** The CRITICAL finding
was that the module-level `pytestmark = pytest.mark.skipif(not _can_run(), ...)`
applied to ALL tests in the module, including the face-level test. If
`_can_run()` was False (regista missing, no Docker/DSN), the entire module
skipped — even with `INTEROP_REQUIRE_FACES=1`. This reintroduced the exact
"green-by-skip" hole Plan 002 was meant to close. The fix: move to per-test
skipif decorators so the face-level test's require logic is independent. I also
fixed the availability probe to check the exact modules the test imports (not a
subset), made the env flag accept common truthy values, and improved the error
message to report which modules are missing.

**What I'm confident in:** The 8-case logic verification (all combinations of
require/faces/can_run) confirms the skip/fail behavior is correct in every
scenario. The CI runs confirm both Linux and Windows pass. The
`--restore-dsn` enforcement is straightforward and well-tested.

**What I'd want a second pair of eyes on:** The Windows CI job is new and
untested beyond "it passed." The windows-latest runner proved agent-suite
imports and 130 unit tests pass on native Windows (4 integration tests skip
cleanly — no Docker/Postgres). But I haven't tested the actual Windows
end-to-end (WI-5.2) — that needs the lab box (mvmcitest01) with a reachable
Postgres and the DPAPI-custody path, which isn't implemented yet (regista
Plan 029 Phase 2).

## Gaps to flag

- **`tests/test_interop.py` `_face_test_should_skip` is computed at import
  time** — if the env var is set after pytest collects the module (e.g., by a
  conftest or plugin), the skip decision is stale. In CI this is fine (env is
  set before pytest runs), but it's a latent inconsistency. Documented in the
  comment, not fixed.
- **`@main` pinning for all three CI git installs** (regista, agent-notes,
  dossier) — a breaking change on any repo's `main` branch turns agent-suite's
  CI red without any change in agent-suite. The plan allows this, but a SHA
  pin would make the suite release reproducible. Low priority but worth noting.
- **The Windows CI job doesn't run the identifier gate** — it's OS-independent
  and already runs on Linux, but for completeness it could be included. Not a
  gap, just a choice.
- **`test_restore_drill.py` still skips locally** (needs `pg_dump`/`psql` CLI
  tools) — it PASSED in CI (CI has those tools), but local dev can't verify it.
  This is a pre-existing condition, not new.
- **Plan 003 Phase 2 / regista Plan 029 Phase 2 interlock** — the DPAPI-custody
  helper needs to be landed once in regista and consumed by both plans. Two
  agents building it twice is the failure mode to avoid. Flagged in the plan
  status and commit message.
