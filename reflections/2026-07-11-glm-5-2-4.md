## On the project

The suite is now past the "what are we building?" phase and into "what are the contracts between the pieces?" The feature matrix (WI-0.1) established the scope; the contract fixtures (WI-0.2) freeze the seams. The key insight is that the suite layer is the natural home for conformance fixtures because it's the one repo that can see all components — even though it doesn't own most of the contracts, it's the layer that composes them.

The architecture is holding up well to the "thin orchestration" constraint. The contract fixtures are JSON data, not Python code, so they don't violate the import-boundary rule. The validation script is stdlib-only and cross-references against agent-suite's own enums (ComponentStatus, Tier, DriftKind, ProjectVerifyStatus, KeyAgeStatus, StoreGrowthStatus) without importing any component. The snapshot assertions for regista/agent-provenance/agent-wake contracts are honestly labeled — they document expected values but can't cross-reference against live code in the lint-and-test CI job because those components aren't installed there.

## On the work done

**What went well:** The fixture structure is clean and consistent — every fixture has contract, version, description, owned_by, consumers, invariants, and domain-specific fields. The cross-referencing approach (import the Python enum, extract .value set, compare against the fixture's declared list) is simple and effective. The adversarial review caught real issues: H2 (missing KeyAgeStatus/StoreGrowthStatus in health contract) was a genuine gap in the health output contract; H3 (corrupted_backup not in the deferred registry) was a real oversight; H1 (false confidence about cross-references) would have been misleading if left unaddressed.

**What was awkward:** The adversarial reviewer correctly identified that 5 of 7 contracts use "snapshot assertions" (hardcoded expected values) rather than live cross-references. This is an inherent limitation — the owning components aren't installed in the lint-and-test CI job. The fix was to be honest about it in the docstrings, which is correct but feels like a half-measure. The real fix would be to add contract cross-reference tests to the interop CI job where regista, agent-notes, and agent-provenance are installed — but that's a larger piece of work that would need to handle the case where a component is absent (skip cleanly).

**What I'm confident in:** The health contract cross-references are solid — all 6 agent-suite enums are validated against live code. The transition referential integrity check (M3) and delivery mode partition check (M2) are straightforward and correct. The subprocess test (M4) ensures the validation script itself works. The deferred mutations guard is now more robust with exact matching (L1 fix) and the corrupted_backup entry (H3).

**What I'd want a second pair of eyes on:** The lifecycle fixture's transition list was hand-transcribed from `regista.canonical_workflow_yaml()`. If regista's canonical workflow changes (new transition, renamed state), the fixture becomes stale and the snapshot assertion would catch it — but only because the expected values are also hardcoded in the test. A real cross-reference would import `regista.canonical_workflow_yaml()` and parse the YAML, which would be more robust but requires regista to be installed.

## Gaps to flag

- **Snapshot assertions vs live cross-references** (`scripts/validate-contracts.py`, `tests/test_contracts.py`): 5 of 7 contracts are validated against hardcoded expected values, not live component code. This is documented in the docstrings but could mislead someone who only reads the CI step name ("Contract conformance check"). The real fix is to add cross-reference tests to the interop CI job.
- **Lifecycle fixture transition list** (`data/contracts/lifecycle.json:15-36`): Hand-transcribed from `regista.canonical_workflow_yaml()`. If regista's canonical workflow changes, the fixture becomes stale. The snapshot assertion catches fixture drift but not code drift.
- **Feature matrix status_source** (`data/v1-feature-matrix.json:4`): Still "hand-assessed". WI-0.3 probe automation is needed to make this reproducible.
- **_DEFERRED_MUTATIONS self-referential guard** (`tests/test_adversarial_corpus.py:978-982`): The `_EXPECTED_DEFERRED` set is defined in the same file as the test. If someone removes an entry from both `_DEFERRED_MUTATIONS` and `_EXPECTED_DEFERRED` in the same commit, the guard passes. This is a known limitation of self-referential guards.
- **Duplicated constants** (`scripts/validate-contracts.py:23-40`, `tests/test_contracts.py:23-40`): `REQUIRED_META_FIELDS` and `EXPECTED_CONTRACTS` are independently defined and could diverge silently. Low priority — they're stable.
