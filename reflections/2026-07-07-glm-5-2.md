## On the project

agent-suite remains a thin orchestration layer that is itself *done* for now —
Plans 004 (dogfood deployment) and 005 (operate the suite) are both "Proposed
2026-07-07" and gated on operator/deployment actions (real Postgres, Vault,
the home-box convergence), not code I can write unilaterally. The interesting
work this session was *in the components*, and it reinforced the blueprint's
core insight: the suite's value is composition, and Windows compatibility is a
property that must be proven in each component, not asserted at the suite
level. agent-suite Plan 003 is the right shape — it audits the gap, records the
posture, and dispatches well-specified fixes to the repos that own the code.

The cross-repo review process worked well. Three different model lineages
(kimi, nemotron-3-ultra, minimax-m3) each caught a distinct class of issue the
implementer (glm) missed: kimi found the security-logic gaps (TOCTOU, errno
contract, mode foot-gun), nemotron found the cross-module consistency bug
(harness.py writing to a different path than config.py reads), and minimax
found the architecture-test evasion and the unfixed sibling anti-pattern
(provenance.py). This is exactly the cross-lineage value the blueprint's §8
single-model-risk section argues for — and it validates that the family's
adversarial-review infrastructure is load-bearing, not ceremonial.

## On the work done

**What went well:** The plan was precise enough that each implementation was a
bounded, well-specified task — three subagents landed green implementations in
parallel. The adversarial reviews were the high-leverage step: every review
returned REQUEST-CHANGES with at least one real finding, and the fixes were
small and targeted. The agent-notes `config_path(home=...)` rebase seam is the
change I'm proudest of — it gives tests a hermetic temp dir while production
uses platformdirs, without duplicating the path logic.

**What I'm confident in:** All three repos pass ruff + mypy + pytest locally.
dossier's `open_no_follow` preserves the atomic `O_NOFOLLOW` kernel guard on
POSIX (no regression) and adds a best-effort symlink pre-check on Windows.
agent-notes' four config/state sites + harness.py now share one platformdirs
resolver. acb's architecture test honestly exempts platformdirs via
`STDLIB_OK` rather than evading it with a lazy import.

**What I'd want a second pair of eyes on:** The Windows CI jobs are added but
unproven (no local windows runner). The dossier symlink test now
skips-if-symlink-creation-fails, which is correct but means a Windows runner
without Developer Mode won't exercise that test at all. The agent-notes
`config_path(home=...)` parameter is a test-injection seam on a production
function — pragmatic, but a slight smell; the alternative (monkeypatching
platformdirs) was rejected for being heavier.

## Gaps to flag

- **agent-notes `harness.py` opencode-config path** (`resolved / ".config" /
  "opencode"`) still hardcodes `~/.config/opencode` — this reads *opencode's*
  config, not agent-notes', so it's correctly out of scope. But on Windows,
  opencode's own config lives elsewhere (`%APPDATA%/opencode`), so this path
  would be wrong there. Pre-existing, not introduced this session. Flag for
  opencode's Windows story, not agent-notes'.
- **acb `adapters.py:185`** same issue — reads opencode's config at a
  hardcoded `~/.config/opencode/opencode.json`. Pre-existing.
- **dossier Windows CI installs regista pinned to `SUITE.lock`** via
  `grep 'ref:' SUITE.lock` — dossier has a `SUITE.lock` (good), but if that
  file drifts or the SHA is unreachable, the Windows job fails opaquely.
  Low-risk but worth noting.
- **agent-notes has no mypy config** — AGENTS.md elsewhere claims "mypy --strict
  in CI from day one" but agent-notes' pyproject has no `[tool.mypy]` and no
  mypy in CI. The platformdirs change introduced zero new mypy errors (522 are
  pre-existing), but the gap itself is a family-convention drift. Not in scope
  to fix this session.
- **The three Windows CI jobs are unverified on actual Windows.** They mirror
  agent-suite's `windows-test` (which passed), but each repo's test surface
  differs. The first push will be the proof.
- **agent-notes testcontainers tests take ~170s** locally (Docker available) —
  the conftest skip-when-Docker-unavailable fix is correct, but the tests are
  slow. Pre-existing, not introduced here.
