## On the project

The regista principal lifecycle is the security-critical boundary for the
entire suite — it's where key registration, proof-of-possession, approval,
and commit happen. The Phase 0 contract foundation (done in a prior session)
was well-designed: frozen dataclasses, versioned digests, closed enums, and
a clean separation between public types and private implementation. Phase 1+2
extends this with DB persistence and atomic commit, which is where the real
complexity lives.

The dossier project is growing from a work tracker into the suite console
(Plan 024). The six-area navigation (Work, Knowledge, Activity, Evidence,
Operations, Administration) is now fully wired with at least initial
providers and pages for each area. The provider pattern is clean — each
area has its own module with frozen dataclasses and read functions that
compose through the gateway. The shell view models (shell.py) provide a
good separation between semantic contracts and visual presentation.

## On the work done

**What went well:** Parallelizing the three implementation streams (regista
lifecycle, dossier knowledge, dossier area providers) across subagents
worked cleanly — no file conflicts because the streams touched different
repos/modules. The adversarial review was extremely valuable: it caught a
critical bug where `record_effective_receipt` treated REJECTED as EFFECTIVE
(a missing `assert_never` dispatch), a non-atomic challenge consumption in
`submit_possession`, and a broken note ID round-trip in the knowledge
provider that would have made every note detail page 404.

**What I'm confident in:** The regista durable lifecycle is architecturally
sound — migration 043 creates proper tables with CHECK constraints, the
commit method is atomic (key registration + signed event + state update in
one transaction), and the idempotency/approval/expiration checks are in
place. The dossier architecture test (test_architecture.py) will catch
any future regression of private imports. The knowledge provider now
correctly filters by entity_kind="note" and uses the proper
`append_event` API with `entity_kind="note"`.

**What needs a second look:** The regista durable path has 19 skipped
tests because there's no PostgreSQL available in this environment. The
DB-level logic (persistence, atomicity, concurrency) is tested only
through the non-durable path and code review — not through actual DB
integration tests. The C3 finding (idempotency check comparing operation
digest instead of intent digest across restarts) was identified but not
fully fixed — it needs a schema change to store the intent digest
separately. The C4 finding (no DB reload path in `_operation`) was also
identified but not fixed — durable operations loaded from DB after a
restart will fail with OPERATION_NOT_FOUND because the in-memory dict is
empty.

## What remains

- **regista C3 (CRITICAL):** Store intent_digest in lifecycle_operations
  and compare against it for idempotency, not the operation digest.
  Currently broken for auto-generated operation_ids across restarts.
- **regista C4 (CRITICAL):** Add `_load_operation_from_db()` fallback in
  `_operation()` when the in-memory lookup misses and durable mode is on.
  Without this, durable operations can't survive a process restart.
- **regista H2 (HIGH):** Handle concurrent INSERT race in
  `_persist_operation` — use `ON CONFLICT DO NOTHING` or `SELECT FOR UPDATE`.
- **regista H3 (HIGH):** Add `SELECT FOR UPDATE` in `record_approval` to
  prevent concurrent approval races.
- **regista H4 (HIGH):** Validate operation_id is a valid UUID when
  provided, since the DB column is UUID type.

## Gaps to flag

- **regista principal_lifecycle.py:** 19 durable tests are skipped (no
  PostgreSQL). The DB-level atomicity, concurrency, and persistence logic
  is untested against a real database. This is the biggest risk.
- **regista principal_lifecycle.py:** The `_operation()` method only checks
  in-memory dict — durable operations loaded from DB after restart will
  fail. This defeats the purpose of durability (C4).
- **regista principal_lifecycle.py:** Idempotency across restarts is broken
  for auto-generated operation_ids (C3). The DB stores the operation digest
  (which includes the UUID), but the idempotency check should use the intent
  digest (which excludes it).
- **dossier knowledge.py:** `create_note` uses `gateway.append_note_event()`
  which calls `self._reg.append_event()` with `entity_kind="note"`. This
  works with real regista but is untested with InMemoryRegista — the
  InMemoryRegista may not support `entity_kind="note"` properly.
- **dossier operations.py:** Accesses `gateway._reg` directly for health
  checks (pool_healthy, maintenance_healthy, regista_version). This couples
  the provider to the gateway's internal implementation.
- **dossier evidence.py:** `read_evidence_summary` calls `gateway.integrity()`
  (full replay) for every project on every page load. This is O(N) full
  replays per request and will be slow with many projects.
- **dossier test_architecture.py:** The string-based checks
  (`test_gateway_does_not_import_custody_or_provision`) can have false
  positives from comments/docstrings mentioning private module names.
  The AST-based check is the reliable one.
- **dossier test_plan015.py:** The `test_inmemory_rotation_works_without_custody`
  test injects a test store mid-test, which is fragile — if the test order
  changes or fixtures are cached, the store injection may not take effect.
